<bind_mount_injection_strategy>
<!-- 关键设计决策：BindMount 注入点 -->

worker 容器创建在 internal/runtime/tasks/worker.go::buildCreateArgs（用 agentapi.HostActionRequest.BindMounts）。
ContainerProxyProvider.PrepareHost 在 worker 容器创建之后才执行（providers 链由 runtime 编排）。

但 sing-box gateway 必须先启动、tun0 必须可用，才能让 worker 内 DNS 查询通过 172.19.0.1 工作。
worker 容器创建时 bind mount 一旦生效，worker 内 DNS 就锁定到 172.19.0.1 —— 但此时 gateway 可能还没起来 → 这是 RESEARCH §Pitfall 2 的风险。

**本 plan 的处理方式**：
1. control-plane 在生成 HostActionRequest 之前，**先**调用 `writeContainerDNSConfig(hostID)` 写源文件到 `<gatewayConfigDir>/resolv.conf` 与 `<gatewayConfigDir>/nsswitch.conf`（这两条路径不依赖 gateway 容器是否启动，只是宿主文件系统操作）。
2. 在 HostActionRequest.BindMounts 里注入两条 ReadOnly bind mount。
3. worker 容器创建后，**默认会走 docker DNS（127.0.0.11）失败 → fallback** —— 不行；docker 容器创建时 resolv.conf 已被 bind mount 接管，没有 fallback。
4. 因此**必须在调用 worker.go::CreateHost 之前** PrepareHost 已经把 gateway 启动好。当前调用链确实是这个顺序：先 PrepareHost（gateway healthy + tun0 监听 172.19.0.1）→ 然后 worker connect to gateway network → 此时 worker 内 DNS 查 172.19.0.1 可达。

**调用链验证**（read-only，由 Task 1 验证现状）：
- runtime/tasks/worker.go 在 docker create worker 之前/之后调用 PrepareHost？
- 如果 worker.go::createHost 先创建容器再调 PrepareHost，bind mount 会让 worker 容器 entrypoint 启动时（在 PrepareHost 之前）DNS 失败。

**Task 1 第一步必须做现状勘察**：grep worker.go 看 PrepareHost / NetworkProvider 调用与 dockerCreate / dockerStart 的相对顺序。
- 如果 PrepareHost 在 docker create 之前 → 安全，按计划注入 BindMount
- 如果 PrepareHost 在 docker start 之后但 docker create 之前 → 通常 worker 容器 entrypoint 在 docker start 时才执行，仍安全
- 如果 PrepareHost 在 docker start 之后 → 不安全，本 plan 需调整顺序（额外 task）

**注入点选择**：在 worker.go::buildCreateArgs 中直接 append 两条 `--mount type=bind,src=<gatewayConfigDir>/resolv.conf,dst=/etc/resolv.conf,readonly`；不依赖 HostActionRequest（避免 control-plane → agent 协议改造）。计算 `gatewayConfigDir(hostID)` 需要把 helper 提升或复制到 worker 包。**推荐**：worker.go 通过 `request.HostID` + 与 network 包共享的常量函数 `network.GatewayConfigDir(hostID)`（导出 helper）拼出路径。
</bind_mount_injection_strategy>

           dir := GatewayConfigDir(hostID)
