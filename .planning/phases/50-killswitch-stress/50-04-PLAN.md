# Phase 50 Plan 04: `docker network disconnect` → worker 不 fallback host bridge (KILL-04)

## Goal

证实「把 gateway 容器从其专属自定义 bridge 网络（`cloudproxy-net-<HostID>`）摘走后，worker 容器内 `curl` 必须失败，且 host eth0 上不出现 worker → 非网关流量。worker **不允许** fallback 到 docker 默认 bridge / host 默认路由直连」。

具体不变量：

- `g.DisconnectGatewayFromBridge(ctx)` 调用后 ≤ 3s，worker `curl --max-time 3 <url>` 必须非 0 退出。
- host eth0 抓包 BPF `src host <workerIP> and not dst host <gatewayIP>` 在 5s 窗口内必须 0 包。
- cleanup：`t.Cleanup` 内 best-effort 调 `g.ReconnectGatewayToBridge(ctx, savedIP)` 恢复；失败仅 t.Logf。

**真相校准**（grep `internal/network/container_proxy_provider.go:323-345`）：
- gateway 容器实际接的是 **专属自定义 bridge** `cloudproxy-net-<HostID>`（`networkName(hostID)` 拼接），**不是** docker 默认 `bridge`。
- worker 容器创建时 `--network bridge`（`internal/runtime/tasks/worker.go:215`），随后通过 `dockerNetworkConnect(ctx, "cloudproxy-net-<HostID>", workerName, "")` 接入同一专属网络。
- KILL-04 「fallback host bridge」的真实含义是：worker 同时挂着 default `bridge` 与 `cloudproxy-net-<HostID>` 两网；disconnect gateway 后，worker 若 fallback 到 default bridge 的默认路由直连外网，host eth0 就会抓到 src=workerIP 的包。

**不在本 plan 范围**：

- SIGKILL / tun0 down / Pumba 注入（属 KILL-01..03）。
- 多自定义 bridge 场景（属未来 phase）。
- disconnect 后自动 reconnect 验证（本 plan 只测「不 fallback」）。

## Steps

1. **grep 锁定 source-of-truth**
   - 网络命名：`networkName(hostID) = "cloudproxy-net-" + hostID`（`internal/network/container_proxy_provider.go:323`）。
   - 网络创建：`dockerNetworkCreate` 走 `docker network create --driver bridge --subnet --gateway`（同文件:357）。
   - gateway 接入：`dockerRunGateway` 用 `--network <netName> --ip <gwIP>`（:371）。
   - worker 接入：`dockerNetworkConnect` 用 `docker network connect [--ip <staticIP>] <netName> <workerName>`（:396）。
   - worker 默认 `--network bridge`（`internal/runtime/tasks/worker.go:215`），所以 disconnect gateway 后，worker 仍在 default bridge 上（理论上能访问外网，除非 nft 兜住）。

2. **`GoldenPath.DisconnectGatewayFromBridge`**
   - 实现：
     ```go
     name, err := p.gatewayDockerName()
     if err != nil { return "", "", fmt.Errorf("disconnect: %w", err) }
     out, _ := exec.CommandContext(ctx, "docker", "inspect", "-f",
         `{{range $k, $v := .NetworkSettings.Networks}}{{$k}}={{$v.IPAddress}};{{end}}`,
         name).Output()
     // 解析 "cloudproxy-net-XXX=10.0.0.2;bridge=172.17.0.5;"
     savedNet, savedIP := pickGatewayBridgeNetwork(string(out))  // 纯函数辅助
     if savedNet == "" { return "", "", errors.New("disconnect: gateway has no cloudproxy-net network") }
     if err := exec.CommandContext(ctx, "docker", "network", "disconnect", savedNet, name).Run(); err != nil {
         return savedNet, savedIP, fmt.Errorf("disconnect %s: %w", savedNet, err)
     }
     return savedNet, savedIP, nil
     ```
   - `pickGatewayBridgeNetwork(raw string) (net, ip string)` 纯函数：
     - 优先取 `cloudproxy-net-` 前缀；
     - 兜底取第一个非空 `name != "bridge"` 的网络；
     - 都没有 → 返回 `("","")`。

3. **`GoldenPath.ReconnectGatewayToBridge`**
   - 实现：
     ```go
     args := []string{"network", "connect"}
     if staticIP != "" { args = append(args, "--ip", staticIP) }
     args = append(args, netName, name)
     return exec.CommandContext(ctx, "docker", args...).Run()
     ```
   - 失败仅 wrap err，调用方在 t.Cleanup 内 t.Logf 容错。

4. **`pickGatewayBridgeNetwork` 纯函数 + 单测**（由 50-shared 引入）：
   - 输入：docker inspect template 输出 `<name>=<ip>;<name>=<ip>;`。
   - 单测 ≥ 4 case：
     - 只有 cloudproxy-net + bridge → 取 cloudproxy-net。
     - 只有 bridge → 取空（兜底分支）。
     - 多个非默认网络 → 取首个 cloudproxy-net 前缀。
     - 空输入 → `("","")`。

5. **e2e 主用例 `TestKillSwitch_04_NetworkDisconnect`**
   - `StartStressGolden(t)`；nil / 句柄缺失 → return / Skip。
   - 总 `ctx, cancel := context.WithTimeout(ctx, 60*time.Second)`。
   - 基线：`g.ProbeOutboundFromUser(ctx, "https://ifconfig.io", 5s)` 必须 exit 0；否则 t.Skipf。
   - 解析 workerIP / gatewayIP（与 KILL-01/02 同流程）。
   - **关键步骤**：
     1. 后台启 `g.TcpdumpOnHostEth0(ctx, "src host <workerIP> and not dst host <gatewayIP>", 5, KillswitchTimingContract.TcpdumpWindow)`。
     2. `savedNet, savedIP, err := g.DisconnectGatewayFromBridge(ctx)`；
        err 含 `gateway has no cloudproxy-net` → t.Skipf "deferred-to-CI: backend gap, gateway not on custom bridge"（**backend GAP** → VERIFICATION 流转 Phase 51）。
        err 非 nil → t.Fatalf。
     3. `t.Cleanup(func() { ... ReconnectGatewayToBridge(savedNet, savedIP) ... })`。
     4. `start := time.Now()`；`probeExit, _ := g.ProbeOutboundFromUser(ctx, probeURL, KillswitchStressContract["KILL-04"].MaxDisconnectMs * time.Millisecond)`，记 `elapsedMs`。
     5. 等 tcpdump goroutine。
     6. `verdict, reason := ClassifyStressResult("KILL-04", StressEvidence{ProbeExitCode: probeExit, LeakedPackets: packets, ElapsedMs: int(elapsedMs)})`。
     7. `verdict != StressVerdictPass` → t.Fatalf 并打印完整时间线 + savedNet / savedIP。
   - cleanup 顺序：`defer cancel()` → `t.Cleanup` 内 Reconnect（注册顺序保证 Reconnect 先于 Scenario.Stop 跑）。

6. **darwin 兜底**
   - 用例文件首行 `//go:build e2e && linux`。
   - `pickGatewayBridgeNetwork` 在 helpers.go 无 tag；darwin `go test ./tests/e2e/ -run "Helpers"` 必须 PASS。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| gateway 不接 cloudproxy-net-* 而是用 macvlan / host 网络（grep 当前真相：bridge driver） | 现状 grep 已确认 `dockerNetworkCreate --driver bridge`，本 plan 直接适配。若未来 backend 切 macvlan，`pickGatewayBridgeNetwork` 返回空 → 用例 t.Skipf + VERIFICATION 标 backend gap 流转 Phase 51（同 Phase 49 LEAK-06/07/08 模式）。 |
| disconnect 后 worker 自动重连（docker daemon 内部行为） | docker network disconnect 不会自动恢复；worker 在 default bridge 上的连接不变，本 plan 测的就是「即便如此 worker 仍不应直连外网」。 |
| cleanup `ReconnectGatewayToBridge` 失败导致 gateway 拿不到原 IP | 用例已保存原 IP；Reconnect 用 `--ip <savedIP>`；同一 subnet 内静态 IP 重新分配应成功。失败仅 t.Logf，每用例独立 GoldenPath 不污染其它用例。 |
| docker inspect template 输出格式漂移（多版本 docker） | `pickGatewayBridgeNetwork` 用 `<name>=<ip>;` 拆分；行为单测 4 case 覆盖；漂移 → darwin 单测立即 fail。 |
| host eth0 tcpdump 在 hosted runner 上不可用 | 同 KILL-01/02：sidecar `--cap-add NET_RAW NET_ADMIN`；不可用 → t.Skip + deferred-to-CI。 |
