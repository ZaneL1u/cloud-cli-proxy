# Phase 50 Plan 02: `ip link set tun0 down` → 容器 curl 失败 (KILL-02)

## Goal

证实「在 gateway 容器内手动 `ip link set tun0 down` 关掉 sing-box 的 tun 设备后，worker 容器内 `curl` 必须在 ≤ 3s 内失败，且 host eth0 上不出现 worker → 非网关流量」。这是 SIGKILL 之外的「软故障」姿势：sing-box 进程没死，但 tun 通道断了，kill-switch 同样必须兜住。

具体不变量：

- `g.SetTunDevDown(ctx)` 调用后 ≤ 3s，worker `curl --max-time 3 <url>` 必须非 0 退出。
- host eth0 抓包 BPF `src host <workerIP> and not dst host <gatewayIP>` 在 5s 窗口内必须 0 包。
- cleanup：`t.Cleanup` 内 best-effort 调 `g.SetTunDevUp(ctx)` 恢复 tun0；失败仅 t.Logf。

**不在本 plan 范围**：

- SIGKILL（属 KILL-01）。
- Pumba netem（属 KILL-03）。
- `docker network disconnect`（属 KILL-04）。
- gateway 进程级 sing-box 重启验证（属未来 phase）。

## Steps

1. **grep 锁定 source-of-truth**
   - tun 设备实名：gateway 容器内 sing-box auto_route 创建的 tun 设备名是 `tun0`（`internal/network/singbox_provider_linux.go:165 wait for tun0`、`internal/network/container_proxy_provider.go:441 docker exec gwName ip link show tun0`）。
   - 注意：worker netns 内 nft `oifname "sb-tun0" accept` 中的 `sb-tun0` 是 **接口名标识**（用于 worker netns nft 规则），与 gateway 容器内 `tun0` 设备名不同；KILL-02 关掉的是 gateway 容器内 `tun0`。
   - 故 `docker exec <gateway> ip link set tun0 down` 是正确入口。

2. **`GoldenPath.SetTunDevDown`**
   - 实现：
     ```go
     name, err := p.gatewayDockerName()
     if err != nil { return fmt.Errorf("set tun0 down: %w", err) }
     cmd := exec.CommandContext(ctx, "docker", "exec", name, "ip", "link", "set", "tun0", "down")
     ```
   - 行为：捕获 stderr；exit 非 0 → wrap 后返回。
   - 不做二次 inspect（`ip link show tun0` 在 down 后输出仍存在，只是 `<NO-CARRIER>`）；行为契约由用例侧 curl + tcpdump 验证。

3. **`GoldenPath.SetTunDevUp`**
   - 与 `SetTunDevDown` 对称，命令 `ip link set tun0 up`。
   - 调用方（用例 cleanup）应包装在 `t.Cleanup(func() { ... })` 里，并用 best-effort 心态：失败仅 `t.Logf`，因为该用例已经测出契约结果，恢复失败只影响后续用例污染（已用每用例独立 GoldenPath 隔离）。

4. **e2e 主用例 `TestKillSwitch_02_TunDevDown`**
   - `StartStressGolden(t)`；nil / host / gateway 句柄缺失 → return / Skip。
   - 总 `ctx, cancel := context.WithTimeout(ctx, 60*time.Second)`。
   - 基线：`g.ProbeOutboundFromUser(ctx, "https://ifconfig.io", 5s)` 必须 exit 0；否则 t.Skipf。
   - 解析 workerIP / gatewayIP（与 KILL-01 同流程）。
   - **关键步骤**：
     1. 后台启 `g.TcpdumpOnHostEth0(ctx, "src host <workerIP> and not dst host <gatewayIP>", 5, KillswitchTimingContract.TcpdumpWindow)`。
     2. `t.Cleanup(func() { upCtx, cancel := context.WithTimeout(...); defer cancel(); if err := g.SetTunDevUp(upCtx); err != nil { t.Logf("cleanup: set tun0 up: %v", err) } })`。
     3. `start := time.Now()`；`g.SetTunDevDown(ctx)` 必须 err == nil；否则 t.Fatalf。
     4. `probeExit, _ := g.ProbeOutboundFromUser(ctx, probeURL, KillswitchStressContract["KILL-02"].MaxDisconnectMs * time.Millisecond)`，记 `elapsedMs`。
     5. 等 tcpdump goroutine。
     6. `verdict, reason := ClassifyStressResult("KILL-02", StressEvidence{ProbeExitCode: probeExit, LeakedPackets: packets, ElapsedMs: int(elapsedMs)})`。
     7. `verdict != StressVerdictPass` → t.Fatalf 并打印完整时间线。
   - cleanup 顺序：`defer cancel()` → `t.Cleanup` 内 SetTunDevUp（注册顺序保证 SetTunDevUp 先于 Scenario.Stop 跑）。

5. **darwin 兜底**
   - 用例文件首行 `//go:build e2e && linux`。
   - `SetTunDevDown` / `SetTunDevUp` 在 `helpers_linux.go`，darwin 不参与编译。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| `ip link set tun0 down` 在 gateway 容器内权限不足（缺 NET_ADMIN） | grep `dockerRunGateway`：gateway 容器创建时已 `--cap-add NET_ADMIN`（`internal/network/container_proxy_provider.go:377`）；ip link 操作天然可用。 |
| tun0 down 之后 sing-box 自动重建 tun 接口 | 这正是要测的：即便 sing-box 试图自愈，**断网窗口期间** worker 不允许有流量直连绕过。用例只检 ≤ 3s 内 curl 失败 + 包数 0，不关心后续恢复。 |
| cleanup `SetTunDevUp` 失败导致 gateway 后续不可用 | 每用例独立 GoldenPath（`tests/e2e/leak/` 范式），cleanup 失败仅污染该 Scenario 实例本身，不会影响其它用例；Scenario.Stop 最终会清掉整套容器。 |
| `docker exec gw ip link set tun0 down` 在容器已 SIGKILL 死掉时失败 | 本 plan 与 KILL-01 串行（CONTEXT §Area 4 决策），开始前 gateway 应仍存活；前置 `ProbeOutboundFromUser` 基线 exit 0 已隐式校验。 |
| tun0 设备名漂移（sing-box 升级后改名） | 锁源码常量：`internal/network/singbox_provider_linux.go:165 / waitForTun0` / `singbox` 配置 `interface_name: "tun0"`；如果未来漂移，darwin 编译期不受影响，CI 真机会直接 fail（设备不存在），驱动同步修改。 |
