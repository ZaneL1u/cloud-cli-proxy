# Phase 48 Plan 02: 容器内 resolv.conf 篡改免疫 (MVS-10)

## Goal

证实「用户在 worker 容器内手工把 `/etc/resolv.conf` 改写为 `nameserver 8.8.8.8` 之后，DNS 解析仍然只走 tun（或被防火墙拒绝），并且 host eth0 抓包看不到 UDP/53 → 8.8.8.8 的直连流量」。

具体到不变量层面：
- 容器内 `cp + cat >` 或 `bash -c 'echo ... > /etc/resolv.conf'` 任一通路尝试改写 resolv.conf：
  - **改写成功**（ro bind mount 在容器侧仍允许 `cat >` 覆盖 inode 内容，因为绕过 mount 表的 user-space 复制路径合法）：随后 `dig` 必须满足「走 tun 返回正常 A 记录」**或**「被防火墙明确 drop / timeout」（复用 Phase 46 `ClassifyDNSResult`）。
  - **改写失败**（系统侧抗住了，譬如 ro bind mount + 文件系统拒绝写）：依然合法（CONTEXT §Area 2 显式声明）。
- 独立 host eth0 抓包 oracle：tcpdump 在窗口内必须捕获 **0** 个 `src host <worker> and udp port 53 and dst 8.8.8.8` 的包，证明没有任何 DNS 直连绕过 tun。

**不在本 plan 范围**：
- DNS-over-TLS / DNS-over-HTTPS 旁路（属 Phase 49 LEAK-02..04）。
- IPv6 DNS 走 53 绕过（属 Phase 49 LEAK-05）。
- 篡改 `/etc/nsswitch.conf` / `LD_PRELOAD` 旁路（属 Phase 49）。

## Steps

1. **grep 锁定 source-of-truth**
   - `internal/network/container_proxy_provider.go::resolvConfContent` —— PrepareGateway 把 `nameserver <tun-side>` 写到 `<DATA_DIR>/gateway/<host_id>/resolv.conf`；docker create 时 `ro bind mount` 到 worker 容器 `/etc/resolv.conf`。
   - `internal/network/dns.go::ConfigureContainerDNS` —— 单独工具函数（写 `/proc/<pid>/root/etc/resolv.conf`），但生产路径走 bind mount，**不**走它。
   - 篡改路径假设：ro bind mount 不能阻止「在 mount point 上覆盖文件内容」（取决于内核版本和 mount flags）。CONTEXT §Area 2 决策接受这一不确定性：`Applied` 与 `Rejected` 都是合法分支。

2. **`ResolvConfTamperResult` 枚举 + 锁定字符串**
   - 在 `helpers.go` 新增：
     ```go
     type ResolvConfTamperResult int

     const (
         TamperUnknown ResolvConfTamperResult = iota
         TamperApplied                 // cat > 成功，resolv.conf 内容已被覆盖
         TamperRejected                // ro / EROFS / EBUSY，写入失败，系统侧抗住
     )

     // ResolvConfTamperContract 锁定本 plan 用于篡改的字面值，避免漂移破坏 bpf filter。
     var ResolvConfTamperContract = struct {
         Nameserver string
     }{Nameserver: "8.8.8.8"}
     ```

3. **`ClassifyResolvConfDNSOutcome` 纯函数**
   - 行为表（来源：CONTEXT §Area 2）：

     | tamper | dnsResult | leakedPackets | ok | reason |
     |---|---|---|---|---|
     | Rejected | * | * | true | "系统侧抗住了 resolv.conf 写入（ro bind mount 生效）" |
     | Applied | Tunneled | 0 | true | "tun 接管 DNS，用户态改写不生效" |
     | Applied | Denied | 0 | true | "防火墙拒绝直连 DNS" |
     | Applied | Leaked | * | false | "DNS 被分类为 Leaked" |
     | Applied | Unknown | * | false | "DNS 分类不明，需 t.Logf stderr" |
     | Applied | Tunneled / Denied | >0 | false | "host eth0 抓到 N 个 UDP/53→8.8.8.8 包，存在直连泄漏" |

   - 单测 6 个分支必须命中。

4. **`GoldenPath.TamperResolvConf`**
   - 实现：
     ```go
     cmd := []string{"bash", "-c", fmt.Sprintf(
         "cp /etc/resolv.conf /tmp/r.bak 2>/dev/null; " +
             "echo 'nameserver %s' > /etc/resolv.conf && " +
             "grep -q '%s' /etc/resolv.conf",
         nameserver, nameserver,
     )}
     code, _, err := workerContainerHandle(p).Exec(ctx, cmd)
     ```
   - `code==0` → `TamperApplied`；`code!=0` → `TamperRejected`（exec 本身错误 → err 非 nil + TamperUnknown）。
   - 不做 atomic rename，保持「用户态最朴素的绕过手法」。

5. **`GoldenPath.ProbeDNSFromUser`**
   - 实现：`dig +short +time=<sec> +tries=1 <domain>`；exit code 0 且 stdout 非空 → Tunneled；其它 → 复用 `ClassifyDNSResult(exitCode, stderr)`。
   - 单源不做 vote，因为本 plan 关心的是 DNS 通路本身，不是回显 IP 是否一致。

6. **e2e 主用例 `TestKillSwitch_ResolvConfTamper_GoldenPath`**
   - `StartGoldenPath(t)`；nil → return。
   - 总 ctx 60s。
   - **基线**：先 `ProbeDNSFromUser(ctx, "example.com", 5s)` 必须返回 `Tunneled`（worker 基线 DNS 正常）；返回 `Denied` 或 `Unknown` → `t.Skip` 避免外网抖动 false-fail。
   - **关键步骤**：
     1. 后台启 `TcpdumpOnHostEth0(ctx, "src host <workerIP> and udp port 53 and dst 8.8.8.8", 5, 5*time.Second)` oracle。
     2. `tamperResult, err := g.TamperResolvConf(ctx, ResolvConfTamperContract.Nameserver)`；err 非 nil → t.Fatalf。
     3. `dnsResult, _ := g.ProbeDNSFromUser(ctx, "example.com", 5*time.Second)`。
     4. 等 tcpdump goroutine 退出，读 `packets`。
     5. `ok, reason := ClassifyResolvConfDNSOutcome(tamperResult, dnsResult, packets)`；`!ok` → `t.Fatalf("MVS-10 fail: %s", reason)`。
   - 把所有中间值（tamperResult / dnsResult / packets / workerIP）打到 `t.Logf` 供 CI artifact 排障。

7. **darwin 兜底**
   - `killswitch_resolvconf_tamper_test.go` 首行 `//go:build e2e && linux`。
   - `helpers.go` 新增片段无 tag。

## Risk & Mitigation

| 风险 | 缓解 |
|------|------|
| ro bind mount 在某些内核版本上彻底拒绝 `cat >` 覆盖 → 用例总是走 Rejected 分支，验证不到 Applied | 这是合法分支（CONTEXT §Area 2），不视作 fail；用例代码把 tamperResult 打到 t.Logf，VERIFICATION 中保留「在两种内核行为下都应 ok」的契约说明。 |
| `dig` 在 worker 容器内未安装 | worker 容器 base image（alpine + busybox / debian）默认含 `dig` 或 `nslookup`；如缺，回退到 `getent hosts example.com`。本 plan 实现按 dig 走，未安装时让 stderr 显示 `dig: not found`，`ClassifyDNSResult` 会归到 `Unknown` → t.Logf 提示用户改用 nslookup。 |
| 篡改后 `dig` 仍命中 8.8.8.8 但因为 nft drop UDP/53→8.8.8.8 直连，timeout 退出 | 这是 `Denied` 分支，按设计 ok=true。 |
| tcpdump sidecar 启动晚于 dig，错过头几个包 | sidecar 通过 `WaitForLog "listening on eth0"` 确认就绪后才返回；`TamperResolvConf` + `ProbeDNSFromUser` 在 sidecar 就绪后才执行。 |
| 篡改后 worker 内 DNS 缓存（systemd-resolved / nscd）造成假阳 | worker image 不预装 systemd / nscd（Phase 45 锁定 alpine base）；dig 默认不走 cache。 |
| 与 Plan 01 共享 `TcpdumpOnHostEth0`，签名漂移会同时打两 plan | 在 helpers.go 中为 `TcpdumpOnHostEth0` 写明确的接口注释；任一漂移会同时编译失败，便于及早发现。 |
